# CTRL前端和Agent选择总结

## 一、当前前端技术栈（已确定）

### 技术栈
- **框架**：React 18 + TypeScript
- **路由**：TanStack Router
- **状态管理**：Zustand + React Query
- **构建工具**：Vite
- **桌面集成**：Tauri 2.0
- **UI动画**：Framer Motion
- **表单验证**：React Hook Form + Zod

### 状态
✅ **技术栈成熟稳定**，需要完善业务功能界面

## 二、Agent框架选择（推荐Hermes）

### 可选框架对比
| 框架 | 优点 | 缺点 | 适合CTRL |
|------|------|------|----------|
| **Hermes** | 自学习、持久化内存、分层安装、相对安全 | 社区较小 | ✅ **推荐** |
| **OpenClaw** | 社区技能丰富、集成广泛 | 较重、安全风险高 | ❌ 不推荐 |
| **Cursor** | 代码生成强、IDE集成 | 仅限于编辑器 | ❌ 不适合 |

### Hermes核心优势
1. **✅ 自学习能力**：从经验创建技能，符合AI自动集成需求
2. **✅ 持久化内存**：跨会话记忆，SQLite存储
3. **✅ 分层安装**：支持6种后端（local/Docker/SSH/Daytona/Singularity/Modal）
4. **✅ 相对安全**：相比OpenClaw安全风险低
5. **✅ MCP支持**：原生支持MCP服务器集成
6. **✅ 中文文档**：有完整中文文档

## 三、分层安装能力（完全支持）

### Hermes支持的6种后端
1. **Local**：本地进程运行
2. **Docker**：容器化部署
3. **SSH**：远程服务器
4. **Daytona**：无服务器后端
5. **Singularity**：高性能计算
6. **Modal**：无服务器函数

### 具体分层方案
```
方案A：完全本地（开发）
CTRL前端 → Hermes本地进程 → 本地工具
优点：数据完全本地
缺点：需要电脑24小时开机

方案B：混合部署（推荐 ✅）
CTRL前端 → API网关 → Hermes云端服务 → 云端工具
                     ↓
             本地知识库 (SQLite/向量DB)
优点：本地知识库 + 云端服务，无需24小时开机

方案C：无服务器（低成本）
CTRL前端 → Cloudflare Workers → Hermes(Modal) → 工具
优点：按需付费，成本低
缺点：冷启动延迟
```

## 四、本地知识库 + 服务器服务（完全可行）

### 架构设计
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  CTRL前端   │    │  API网关    │    │ Hermes服务  │
│  (Tauri)    │────│ (Cloudflare)│────│  (云端)     │
└─────────────┘    └─────────────┘    └─────────────┘
        │                   │                   │
        │                   │                   │
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 本地知识库  │    │  工具执行   │    │  MCP服务器  │
│ (SQLite)    │    │ (CLI/HTTP)  │    │  (云端/本地)│
└─────────────┘    └─────────────┘    └─────────────┘
```

### 关键特性
1. **✅ 本地知识库**：SQLite + 向量搜索，数据隐私保护
2. **✅ 云端服务**：Hermes运行在服务器，无需本地24小时开机
3. **✅ 数据同步**：增量加密同步
4. **✅ 成本可控**：Cloudflare免费层 + 按需付费
