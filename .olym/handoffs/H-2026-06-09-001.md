## 现象

CTRL 当前的架构是 Pi 单脑（coding 模式），但 Irisy 需要同时支持助理和编码两个角色。
bao 认为助手和编码不能用同一个 brain 切换模式（2026-06-09 session 结论：两条不同的 pipe）。

**关键发现**：
1. opencode 的 coding 能力远强于 Pi（LSP + formatter + symbol search + plan agent）
2. Hermes 有长期记忆（RAG + embeddings），但 opencode 没有
3. CTRL 的 coding 能力还不足（需要 LSP + formatter + symbol search）

## 已确定的方向

1. **两个对等的 agent 进程**，不是嵌套关系
   - opencode（编码）— 新加，跑 `opencode serve`（headless）
   - Hermes（助理）— 新加，跑 `hermes mcp serve`
2. **各自的 provider 各自管**
   - opencode → `~/.local/share/opencode/auth.json` 自己管
   - Hermes → `~/.hermes/config.yaml` 自己管
3. **opencode 不通过 Extension API 接 CTRL**
   - opencode Extension API 未知
   - 新方案：opencode HTTP API，CTRL 做 HTTP client
4. **Hermes 不通过 HTTP proxy 接 CTRL**
   - 上次集成失败的原因：streamSimple 没有 tools 字段，静默破坏了 tool calling
   - 新方案：Hermes 直接 `hermes mcp serve` 做 MCP server，CTRL 做 MCP client
5. **opencode 的 coding 能力远强于 Pi**
   - LSP integration（代码理解）
   - formatter（代码格式化）
   - symbol search（符号搜索）
   - plan agent（任务规划）
   - summary agent（对话摘要）
6. **Hermes 有长期记忆**
   - RAG + embeddings
   - 对话能力强
7. **CTRL 前端已有多面板基础设施**
   - `WorkspaceInstance` + `TabBar` + `InstanceSwitcher`
   - `IrisyChat` 组件可复用（无 iframe fallback）
8. **两个 brain 之间不共享上下文**（确认 2026-06-09）
9. **Hermes 前端渲染使用原生 React 组件**（确认 2026-06-09，无 iframe fallback）

## 讨论 / 备注

### 为什么选 opencode 而不是 Pi

**opencode 的 coding 能力远强于 Pi**：
1. LSP integration（代码理解）
2. formatter（代码格式化）
3. symbol search（符号搜索）
4. plan agent（任务规划）
5. summary agent（对话摘要）
6. title agent（自动生成标题）
7. HTTP API 完整（OpenAPI 规范生成）

**Pi 的局限**：
1. 无 LSP（无法理解代码结构）
2. 无 formatter（无法格式化代码）
3. 无 symbol search（无法搜索符号）
4. 无 plan agent（无法做复杂任务规划）
5. 无 summary agent（无法做长对话摘要）

### 为什么选 Hermes 而不是 opencode 做 assistant

**opencode 没有长期记忆**：
1. 无 RAG（无法检索 vault）
2. 无 embeddings（无法向量化）
3. 无法基于 CTRL vault 回答问题

**Hermes 有长期记忆**：
1. RAG + embeddings
2. 对话能力强（对话 assistant）
3. 工具生态够用（对话 + 知识检索）

### 为什么不选 CTRL (Irisy) 做 assistant

**Irisy 的局限**：
1. 无 plan agent（无法做复杂任务规划）
2. 无 summary agent（无法做长对话摘要）
3. session 管理弱（只有基础 session）

**Hermes 的优势**：
1. 对话能力强（对话 assistant）
2. RAG + embeddings（长期记忆）
3. 工具生态够用（对话 + 知识检索）

### Hermes 之前集成失败（PR #62）的 6 个根因
1. Hermes 和 Pi 是同级竞争者，不是互补
2. 两个 agent runtime 打架
3. HTTP proxy 破坏 tool calling
4. hermes3:8b 本地模型太弱
5. 记忆优势 CTRL 已有（vault/irisy/SOUL.md）
6. bao 说"清除所有键帽"

本次方案不同：两个对等进程，各接各的 MCP/HTTP，CTRL 不写中间层。

### opencode 的风险
1. opencode Extension API 未知（无法注入 capability segments）
2. opencode Credential 明文存储（`~/.local/share/opencode/auth.json`）
3. opencode HTTP server 复杂度（需要 spawn + port 管理）

缓解：
1. Extension API → 不依赖（opencode 只做 coding，不需要 extension）
2. Credential → 集成到 macOS Keychain（不用 opencode 的明文 JSON）
3. HTTP server → KernelSupervisor 统一管理（spawn + port 管理）

### Hermes 的风险
1. Hermes 集成复杂（MCP server）
2. Hermes 无 session 管理（但够用）
3. Hermes 无规划/摘要能力（但 assistant 不需要）

缓解：
1. MCP server → 使用 stdio MCP client（简单）
2. session 管理 → 基础 session 够用
3. 规划/摘要 → assistant 不需要（对话 + RAG 足够）

### 数据流

#### Assistant (Hermes) 路径

```
User types in Assistant Tab (IrisyChat)
  ↓
PWA → Tauri invoke `hermes_chat_stream`
  ↓
Kernel HermesClient → Hermes subprocess (stdio MCP)
  ↓
Hermes internal loop (LLM → MCP tools → loop)
  ↓
Hermes tools: vault.*, clipboard.*, search, etc.
  ↓
Tauri event `hermes-chat-delta` → PWA streaming
```

#### Coding (opencode) 路径

```
User types in Coding Tab (OpencodeChat)
  ↓
PWA → Tauri invoke `opencode_chat_stream`
  ↓
Kernel OpencodeClient → opencode HTTP server
  ↓
opencode internal loop (LLM → tools → loop)
  ↓
opencode tools: file.*, pty.*, session.*, mcp.*, lsp.*, formatter.*
  ↓
Tauri event `opencode-chat-delta` → PWA streaming
```

### 实现路径

#### Phase 1: Kernel Opencode HTTP Client (P0)

1. **Opencode Spawner** (`src-tauri/src/kernel/opencode_supervisor.rs`)
   - 新增 `OpencodeSupervisor` struct
   - 在 `KernelSupervisor::start` 中 spawn opencode subprocess
   - 命令: `opencode serve --hostname 127.0.0.1 --port 0`（随机 port）
   - 健康检查: subprocess alive + HTTP ping
   - 从 STDOUT/STDERR 解析随机 port

2. **Opencode HTTP Client** (`src-tauri/src/kernel/opencode_client.rs`)
   - 基于 HTTP client（reqwest 或 ureq）
   - 连接到 opencode HTTP server
   - 调用 opencode API: `session_prompt`, `session_messages`, `session_create`
   - Streaming: Server-Sent Events (SSE)

3. **Tauri Command** (`src-tauri/src/commands/opencode_chat.rs`)
   - `#[tauri::command] opencode_chat_stream`
   - 参数: `sessionID: Option<String>, message: String`
   - 返回: streaming delta via Tauri event `opencode-chat-delta`

#### Phase 2: Kernel Hermes MCP Client (P0)

1. **Hermes Spawner** (`src-tauri/src/kernel/hermes_supervisor.rs`)
   - 新增 `HermesSupervisor` struct
   - 在 `KernelSupervisor::start` 中 spawn Hermes subprocess
   - 命令: `hermes mcp serve` (stdio mode)
   - 健康检查: subprocess alive + stdio MCP handshake

2. **Hermes MCP Client** (`src-tauri/src/kernel/hermes_client.rs`)
   - 基于 Anthropic rmcp SDK (已有用于 mcp_host)
   - stdio-based MCP client 连接到 Hermes subprocess
   - 注册 MCP tools 到 kernel: `mcp.hermes_chat`
   - 工具调用: hermes 的 MCP server 提供的工具列表

3. **Tauri Command** (`src-tauri/src/commands/hermes_chat.rs`)
   - `#[tauri::command] hermes_chat_stream`
   - 参数: `messages: Vec<ChatMessage>`
   - 返回: streaming delta via Tauri event `hermes-chat-delta`

#### Phase 3: PWA Coding Tab (P0)

1. **Routing** (`packages/ctrl-web/src/routes/coding.tsx`)
   - 替换现有 Pi coding tab
