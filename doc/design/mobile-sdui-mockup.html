02 一个 part 怎么来的(协议)

包的 describe 回一个 Surface;手机 parts.map(p => registry[p.kind]) 渲染;点动作走同一条 gate 隧道。

1 · describe
包自报界面返回 Surface = 有序 typed parts(kind + data + actions),就是 §14 describe 多回一个形状。
→
2 · query
灌数据每个 part 的 data 从桌面 kernel 读(local-is-truth),经 relay 隧道到手机。
→
3 · registry
通用渲染手机 registry[p.kind] 挑组件;不认识的 kind → 降级成 JSON,不崩。
→
4 · produce
动作回流点「刷新/新建」→ 声明的 gate 调用(经隧道 + 审计)→ 重灌 → 重渲染。客户端不含逻辑。
// 包 describe 回的 Surface —— 手机拿到就能渲,零专属代码 { v:1, pack:"crm", title:"客户", parts:[ { kind:"record", id:"overview", data:{ 新增客户:18, 待跟进:7, 本月成交:"¥42,800" } }, { kind:"table", id:"recent", data:{ columns:[...], rows:[...] } }, { kind:"list", id:"todo", actions:[ { id:"new", label:"新建客户", verb:"produce", source:"crm", op:"contact.create" } ] } ]}

03 前端显示上,这带来什么

✓ 好处(前端)
  • 任何包立即上手机 —— 装了新包,describe 一回,手机自动出 tab + 渲染,不发版
  • 一致的观感 —— 都走同一套 PartKind 组件(卡片/表/record/list),手机上不会各包各样式
  • 桌面 = 手机同一份 UI —— 同一个 registry,桌面预览和手机远程是同一个渲染
  • 降级不崩 —— 老手机遇到新 kind → 显示 JSON,不白屏
✗ 边界(诚实)
  • 不是像素级自由 —— 包只能用 registry 里有的 kind 组合;要全新视觉得先加一个 PartKind(一次,全包共享)
  • 竖排 parts,非任意布局 —— 手机就是卡片竖滚;复杂布局引擎不做(过度)
  • 股票现有的富卡 —— 作为 stock 这个 kind 保留;其余包用通用 kind,想要专属富卡就升级成新 kind
