## §1 Mcp execution — MCP outward / Actor inward

**Outward protocol = MCP** (Anthropic Model Context Protocol). **Inward runtime = Actor** (one of ADR-001 §2's 5 primitives).

- **MCPServerActor** = well-known `Actor` subclass wrapping an MCP server. Not a 6th primitive.
- **4 mcp sources** (ADR-001 §3 v9) — MCP / OAuth / Local agent / Builtin — all express uniformly as MCP servers. ST-SS is retired as a source; implementation differences are absorbed in MCPServerActor subclasses (SubprocessActor for Pattern B/C/D, OAuthCapability for Pattern E).
- **v1 sandbox** = OS-level (macOS sandbox-exec / Linux landlock+seccomp / Win AppContainer) with capability gates declared in manifest. Missing capability → syscall block.
- **WASM** demoted from default to optional high-security actor impl. v1.x re-evaluation = WasmEdge if needed.

Day-1 compatibility with Anthropic MCP ecosystem (10K+ servers). Irisy / Claude Code / Cursor / any MCP host invokes mcps with no integration code. Creators write MCP once, runs everywhere.

## §2 Tauri 2 updater + three-mirror channel

Self-update = **Tauri 2 native updater** + **three-mirror endpoint array (ordered fallback)** + **three channels (stable / beta / dev)** + **per-user install (no UAC)** + **signing key in macOS keychain**.

**Mirror priority order**:
1. Tokyo Caddy (`updates.ctrlapplab.com`)
2. CF Worker + R2 (`ctrl-updates.workers.dev`)
3. GitHub Releases

**Channel switching**: `~/.ctrl/state/update-channel` file (PWA Settings toggle). URL embeds channel segment.

**Signing**: private key in macOS keychain `tauri-sign / ctrl-updater`; public key in `src-tauri/tauri.conf.json` plugins.updater.pubkey. Mandatory redundant backup (1Password + age-encrypted disk).

**v1 ship reality**: single-mirror `soodooi/CTRL-releases` only. 3-mirror failover deferred to v1.1 (gated on CN P95 SLO + bao budget for Win OV signing ~$200-400/yr).

## §3 4-layer auto-update strategy

Each layer self-driving (no manual reinstall). Memory `feedback_always_use_upgrade_path` 🔒 — bao 2026-05-26: 新版本必须 Tauri auto-updater 分发, 手动 cp -R 破坏 AX 权限 (hotkey 失效根因).

| # | Layer | Mechanism | Trigger | Cadence |
|---|---|---|---|---|
| 1 | **CTRL.app binary** | Tauri 2 updater + mirror (§2) | App launch + opportunistic poll | weekly stable / nightly dev |
| 2 | **External agents (hermes / opencode / kairo)** | Single pin SSOT bump, gated by the ACP contract probe + L3 approval (ADR-002 §1.8.4 upgrade规范) | New upstream release → probe → suggest | per agent release, user-approved (patch may earn L4) |
| 3 | **Mcp upstream** | Per-mcp manifest `upstream_url` + signature; kernel polls + applies per user tier (§4) | Daily background OR Irisy "improvements available" bubble | per-mcp |
| 4 | **PWA bundle** | `vite-plugin-pwa` Service Worker auto-update + reload prompt | App start + visibility-change | per CTRL.app deploy (coupled with Layer 1) |

Layer 2 was "Pi (brain)" through v1; Pi exited the hot path (ADR-002 v19+), so Layer 2 is now the **external-agent tier** — hermes / opencode / kairo, each pin-bumped under the ADR-002 §1.8.4 upgrade规范 (single pin SSOT + ACP contract probe + L3 gate). This is the rollout home for the agent-upgrade discipline.

## §4 3-tier user-adjustment model (applies to Layer 3 mcps)

Memory `decision_keycap_3_tier_adjustment` 🔒.

| Tier | What user did | Update strategy | Conflict UX |
|---|---|---|---|
| **Config** | Filled `manifest.config_schema` fields (API keys, defaults, paths). No code/manifest body edits | Clean overwrite. User's `config` namespace migrates by `config_schema` version; renames via `config_migration` declarations | None — config is forward-compatible by design |
| **Patch** | Overrode manifest fields or added per-mcp override layer (via Irisy Improvement stage, ADR-005 § 8-stage) | Smart 3-way merge: upstream + user patch + base. Conflicts → Irisy bubble at Debug stage; user accepts upstream / keeps patch / merges manually | Irisy presents diff + suggests resolution |
| **Fork** | Chose Fork at install OR explicitly disconnected from upstream. Personal branch | No auto-update. Irisy occasionally prompts "upstream has new features X, Y — want to cherry-pick?" | User-driven, explicit cherry-pick |

User-tier picker UI lives in Settings → per-mcp "update behavior" + Irisy Config stage on first install.

## §5 Cross-layer coordination

- Layers update INDEPENDENTLY by default
- Compatibility checked via Schema/API version pinned in manifests
- CTRL.app declares supported mcp manifest schema range; mcp declares required CTRL.app min version
- Mismatch → mcp shown as "needs CTRL.app upgrade" in Pool; not invoked until resolved
- User-facing UX = "CTRL is up to date" / "1 mcp update available" / "tap to install" — never "rebuild from source"

## §6 Signature + integrity (Layers 2-3)

- Layer 2 (external agents): pinned upstream package/artifact + ACP contract probe + L3 approval (ADR-002 §1.8.4)
- ~~Layer 3 (mcp): per-mcp signing key (creator-supplied), public key in `manifest.signing.pubkey`; kernel rejects update if signature fails~~ **SUPERSEDED v3 — this was the key-swap anti-pattern.** A verification key carried *inside the bundle it verifies* lets an attacker re-sign with their own key + replace the embedded pubkey, so verification always passes (NIST CSWP code-signing; US Pat. 7,552,342). The key must live OUTSIDE the bundle.

### §6.1 Pack supply-chain trust model (v3, 2026-06-26 — red-team C1 supply chain)

Research (`vault/ctrl/research/pack-supply-chain-trust.md`, 2026-06-26): mature mechanisms were scored against CTRL's hard constraints (no account, local-first, multi-source, `.mcpb` zip, plain-text). **Sigstore keyless** needs an OIDC *account* (Google/GitHub) → rejected. **Full TUF** needs a live online metadata/timestamp pipeline per source → too heavy for v1 + third-party authors. **npm provenance** needs central registry + cloud CI → rejected. **Registry signing root** = a central trust locus CTRL avoids. The `.mcpb` spec itself carries **no** signature/hash field, and the MCP Registry signs *nothing* (verifies namespace identity only). Closest live analog = **Obsidian community plugins** (local-first, third-party-dominant, no enforceable sandbox, default-off Restricted Mode + author-hosted releases).

**Decision: minisign/signify-style detached Ed25519 + TOFU key-pinning + mandatory sha256 hash-pin + default-deny install gate.** Account-free, offline, multi-source-uniform, `vim`/`git diff`-inspectable, uses the unsigned `.mcpb` as-is (signature is a sidecar, no format fork). Borrows TUF *ideas* (per-source root, forward-chained key rotation, threshold for high-value packs) without running TUF's servers.

**v1 (no registry) — install-time verification:**
1. **[SHIPPED v3] sha256 hash-pin** — `install_mcpb_blocking` hashes the bundle before trusting it + writes `~/.ctrl/mcps/<id>/installed.json {bundle_sha256}` (git-diffable). Closes the "unzip with zero verification" gap. Integrity, not authorship.
2. **Detached Ed25519 signature** over the bundle hash (minisign/signify-compatible, sidecar file; trusted-comment = pack+version+author fingerprint). **Verification key NEVER inside the `.mcpb`** — author publishes pubkey out-of-band (repo root `minisign.pub`, ≥2 channels).
3. **TOFU key-pin** — pin author fingerprint on first install (`~/Documents/CTRL/.trust/packs.toml`, signed by the user's Keychain identity so a malicious pack can't edit it); same key (or forward-chained successor) required on every update; different unsigned key → block + plain-text old-vs-new fingerprint warning.
4. **Default-deny gate** before any pack code runs (Obsidian Restricted Mode): a fresh pack is inert until the user approves; approval screen shows fingerprint + hash + (for MCP packs) the *full* tool descriptions the model will see — defeats MCP tool-poisoning / rug-pull (Invariant Labs / Trail of Bits: re-hash tool descriptions each server start, diff vs the approved pin).

**Revocation (v1, no server):** forward-chained key rotation (`next_pubkey`, signify model) → rotate with no revocation server; local kill via TOFU pin mismatch; opportunistic **CTRL-signed** `revoked-hashes` denylist (the one justified CTRL-held key — for CTRL's own release artifacts, not user identity).

**v1.1 (with MCP Registry / `.well-known`):** adopt registry namespace identity (`.well-known/mcp-registry-auth`, GitHub OIDC trusted-publishing) as a *second* trust signal that shrinks the first-contact window — NOT a replacement for CTRL's own sig+hash (registry signs/scans nothing). Graduate first-party `ctrl-<name>` packs to TUF-style threshold + offline root.

**Residual (state plainly):** first-install/first-contact trust is irreducible without a central PKI (TOFU + wide out-of-band key dispersion mitigates, doesn't cure); TOFU does not stop a *legitimately-signed* malicious update (→ default-deny gate + no-sandbox honesty); signing is integrity+continuity, not content-safety (pair with the `:17873` gate + OS sandbox for runtime containment).

## Release Acceptance (§2)

These criteria are the bounded repository contract enforced before shipping; the broader design backlog below remains visible in soft audits but does not permanently block an otherwise valid release.

- [x] `scripts/release.sh` requires one clean committed source tree with synchronized root/web/Tauri/Cargo versions and re-verifies that tree before publication.
- [x] Every automatic governance baseline is strictly older than the target and resolves through a remote immutable source tag plus exact, non-conflicting public body/`latest.json`/`source-provenance.json` evidence; legacy bootstrap requires explicit one-time approval, an ancestral non-HEAD source, and a demonstrably legacy `latest.json` before tag or sidecar mutation.
- [x] Release retries treat the updater archive plus `latest.json` as one signed build pair: source commit/tag, version, exact URL, SHA-256, and Tauri/minisign signature are re-verified from downloaded public assets; incomplete or invalid pairs are replaced together.
- [x] Published macOS release bundles are produced through `scripts/release.sh` with the configured stable identity; the script removes stale output and verifies the fresh app's strict code signature, Authority, identifier, certificate-bound Designated Requirement, and absence of ad-hoc/`cdhash` identity.
- [x] Strict release audit checks this explicit Release Acceptance scope; soft development/CI audit continues to report all design Acceptance debt.
